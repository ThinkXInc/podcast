# 運用コマンド手順（原本）

> ユーザーの運用マニュアル原文を、サイト名を `<site>` に一般化して残したもの。
> 実際のサイトでは `<site>` を各サイト名に読み替える（このプロジェクトでは `podcast`）。
> Claude Code 向けの運用ルールは `OPS_GENERAL.md` にあり、詳細手順としてこのファイルを参照する。
> 見出しは参照用。コマンドはユーザー原文の形をそのまま踏襲。

## Build & Run App

### build watcher - front end app 🪐
フロントエンド（views）のビルド監視を起動する。
```
cd /src/<site>/web-server
cd views
. ./watchappviews.sh
```

### build watcher - simplicity 🪐（simplicity を使うサイトのみ）
```
cd /src/<site>/web-server
cd views
. ./watchsimplicity.sh
```

### uwsgi log 🌃【Summary】UWSGI Commands 🪐
uWSGI のログを追う。
```
sudo journalctl -fu uwsgi_<site>.service --output cat -n 3000
```

### restart uwsgi 🪐
uWSGI を再起動する（コード変更の反映など）。方式はサイトによる。

systemctl 直接:
```
cd /src/<site>/web-server
sudo systemctl restart uwsgi_<site>.service
```
restart.sh 方式（etc/restart.sh を持つサイト）:
```
cd /src/<site>/web-server
. ./etc/restart.sh uwsgi
```

## Logs

### uWSGI ログ（複数の見方）
follow（追従）:
```
sudo journalctl -fu uwsgi_<site>.service --output cat -n 3000
```
過去分をページャで:
```
sudo journalctl -u uwsgi_<site>.service --no-pager | tail -n 3000 | less
```
直近だけ:
```
sudo journalctl -u uwsgi_<site>.service -n 700
```
screen 方式（`.screenrc_uwsgi` を持つサイト）:
```
cd /src/<site>/web-server/uwsgi
screen -c .screenrc_uwsgi
```

### nginx ログ（screen 方式）
```
cd /src/<site>/web-server/nginx
screen -c .screenrc_nginx
```

## nginx（基盤は nginx-root、サイト設定は各リポジトリ内）
サイトの server 設定は各リポジトリ内 `web-server/nginx/conf.d/`（podcast は `web/nginx/conf.d/`）。
nginx 基盤（nginx.conf 等）は別リポジトリ nginx-root。

### 設定の文法チェック
```
sudo nginx -t -c /src/nginx-root/nginx.conf
```

### nginx 再読み込み（設定変更の反映。プロセスは落とさない）
nginx-root 経由:
```
sudo nginx -s reload -c /src/nginx-root/nginx.conf
```
restart.sh 方式（nginx/restart.sh を持つサイト）:
```
cd /src/<site>/web-server/nginx
. ./restart.sh reload
```

### 単独ホスト（nginx-root なしで、そのサイトだけ nginx で確認したいとき）
各サイトの conf は自己完結しているので、そのサイト単体でも nginx に読ませられる。
```
sudo nginx -t -c /src/<site>/web-server/nginx/nginx.conf   # サイト内に nginx.conf を持つ場合
```

## サイト固有の運用について
上記は**全サイト共通**の運用（build / uWSGI / log / nginx）。これに対し、各サイト独自の
機能（例: quantz-web の VectorDB / Celery / MongoDB / Redis / accessdb / billing_scheduler /
crontab 等）は**そのサイトのリポジトリ内の固有マニュアル**に置く。この共通原本には入れない
（サイトに無い手順を混ぜると煩雑になるため）。podcast 固有の運用があれば podcast 側に追記する。
